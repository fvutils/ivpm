# FuseSoC Integration Design

## Overview

[FuseSoC](https://github.com/olofk/fusesoc) is a package manager and build
system for FPGA/ASIC HDL (Verilog/VHDL) projects. It uses `.core` files
(CAPI=2: YAML) to describe hardware IP cores, their source file sets,
dependencies (by VLNV address), and build targets.

[IVPM](https://github.com/fvutils/ivpm) is a multi-format package manager
that fetches software and hardware dependencies into a project-local `packages/`
directory.

This document describes the integration strategy that lets IVPM add value to
projects that use FuseSoC.

---

## Problem Statement

FuseSoC must be told where to find cores. Today, a project using both tools
must:

1. Declare HDL dependencies in `ivpm.yaml` as raw git URLs.
2. Manually maintain a `fusesoc.conf` listing each fetched package directory.
3. Keep these in sync whenever dependencies change.

The integration goal: after `ivpm update`, FuseSoC can immediately find all
fetched HDL cores — with no manual configuration file maintenance.

---

## Core Concepts

### FuseSoC VLNV Addressing

FuseSoC identifies cores by a four-part **VLNV** (Vendor:Library:Name:Version)
string, e.g. `::wb_intercon:1.0`. The public registry lives at
[github.com/fusesoc/fusesoc-cores](https://github.com/fusesoc/fusesoc-cores).
Each entry is a `.core` file with a `provider:` block containing the git URL
and version tag needed to fetch the actual source.

### IVPM Package Handlers

IVPM's `PackageHandler` subclasses implement lifecycle hooks:

| Hook | When called |
|---|---|
| `on_root_pre_load(update_info)` | Before any packages are fetched (main thread) |
| `on_leaf_post_load(pkg, update_info)` | After each package is ready on disk (parallel) |
| `on_root_post_load(update_info)` | After all packages are ready (main thread) |
| `get_state_entries()` | For persistence in `ivpm.json["handlers"]` |

Handlers are registered via `ivpm.handlers` entry points and activated when
`package.with.<name>:` appears in `ivpm.yaml`. Handlers run in `phase` order
during `on_root_post_load`.

### IVPM direnv Handler

The existing `PackageHandlerDirenv` (phase=0) collects `export.envrc` files
from each fetched package and generates `packages/packages.envrc` — a single
sourceable file that sets up all package environments. The project `.envrc`
typically contains `source_env packages/packages.envrc`.

---

## Proposed Integration

The integration has **two independent, composable parts**:

| Part | Purpose |
|---|---|
| **`PackageHandlerFuseSoC`** | Discovers `.core` dirs from deps, generates three output artifacts, integrates with the direnv handler |
| **`fusesoc` source type** *(optional)* | Resolves VLNV identifiers against the fusesoc-cores registry and fetches the underlying git repo |

---

## Part 1 — `PackageHandlerFuseSoC` (New Handler)

### Core Directory Discovery — Publish / Import Pattern

Modelled on the agents handler's skill-discovery pattern.

#### Publishing (per dependency)

A dependency advertises which directories contain its FuseSoC cores in two ways:

**Auto-detection (no annotation needed):**
The handler scans the package root (non-recursively) for `*.core` files.
If any are found the package root is treated as a core directory.

**Explicit declaration** in the dependency's own `ivpm.yaml`:
```yaml
package:
  name: wb_intercon
  with:
    fusesoc:
      cores:
        - cores/          # directory containing .core files
        - extra.core      # or a specific .core file (dir is registered)
```
Glob patterns are supported (e.g. `ip/*/rtl`). Explicit declarations
**replace** auto-detection for that package.

#### Importing (root project)

The root project controls which packages' cores are aggregated via
`with.fusesoc:` in its `ivpm.yaml`:

```yaml
package:
  name: my_soc
  with:
    fusesoc:
      # Optional: limit imports to named packages (default: all)
      import: all           # or a list: [wb_intercon, uart16550]

      # Optional: update fusesoc.conf in the project root (default: false)
      update-conf: false
```

The root project's own `.core` files (in the project directory itself) are
always included if any exist.

### Generated Outputs

All artifacts are written inside `<deps-dir>/` so they are gitignored by
default (same as all IVPM-fetched content).

#### 1. `packages/fusesoc-cores.txt`

Machine-readable list of discovered core directories, one per line, relative
to the project root. Intended for scripting and tooling that needs to enumerate
available cores without parsing shell syntax.

```
packages/wb_intercon
packages/uart16550/cores
.
```

#### 2. `packages/fusesoc-cores.envrc`

Shell-sourceable file setting `FUSESOC_CORES`:

```bash
# Generated by IVPM fusesoc handler — do not edit manually
export FUSESOC_CORES="<abs-path>/packages/wb_intercon:<abs-path>/packages/uart16550/cores:."
```

This file is the primary artifact. It can be sourced directly in a project
`.envrc`, used in CI, or referenced from any shell script.

#### 3. Contribution to `packages/packages.envrc`

When the `direnv` handler is also active it generates `packages/packages.envrc`
at `phase=0`. The FuseSoC handler runs at `phase=10` and **appends** a single
`source_env` line to that file:

```bash
# fusesoc cores (appended by IVPM fusesoc handler)
source_env ./fusesoc-cores.envrc
```

This means a project that already uses the direnv handler gets `FUSESOC_CORES`
set automatically, with no additional lines needed in the project `.envrc`.

If the direnv handler is not active, `packages/fusesoc-cores.envrc` is still
generated and the user can source it manually or in CI.

#### 4. Optional: `fusesoc.conf` update (opt-in)

When `update-conf: true` is set, the handler also writes IVPM-managed sections
to the project `fusesoc.conf`. Only sections prefixed `library.ivpm.` are
touched; all user-maintained sections are preserved.

```ini
# fusesoc.conf (user-maintained sections unchanged)

[library.ivpm.wb_intercon]
location = /abs/path/to/packages/wb_intercon
sync-type = local

[library.ivpm.uart16550]
location = /abs/path/to/packages/uart16550/cores
sync-type = local
```

> **Note:** Python's `configparser` (used by FuseSoC) has no include/reference
> mechanism, so there is no way to point from `fusesoc.conf` to a file
> generated in `packages/`. The `.envrc` path (above) is the recommended
> approach. The `fusesoc.conf` update is provided for users who cannot use
> `.envrc`.

### Activation

Bare `with: fusesoc:` is sufficient:

```yaml
package:
  name: my_soc
  with:
    fusesoc:                 # activate handler, all defaults
```

All options under `with.fusesoc:` at the root project level:

| Key | Default | Description |
|---|---|---|
| `import` | `all` | Package names to import cores from, or `all` |
| `update-conf` | `false` | Patch `fusesoc.conf` with IVPM-managed library sections |

### Handler Implementation Sketch

```python
@dc.dataclass
class PackageHandlerFuseSoC(PackageHandler):
    name               = "fusesoc"
    description        = "Discovers .core dirs from deps; generates fusesoc-cores.envrc and fusesoc-cores.txt"
    phase              = 10   # after direnv (phase=0)
    leaf_when          = None
    root_when          = None

    # pkg_name -> list of absolute core dir paths
    _core_dirs: Dict[str, List[str]] = dc.field(default_factory=dict)

    def reset(self):
        self._core_dirs = {}

    def on_leaf_post_load(self, pkg: Package, update_info: ProjectUpdateInfo):
        if not hasattr(pkg, "path") or pkg.path is None:
            return
        if getattr(pkg, "src_type", None) == "pypi":
            return

        dirs = self._get_core_dirs(pkg)
        if dirs:
            with self._lock:
                self._core_dirs[pkg.name] = dirs

    def on_root_post_load(self, update_info: ProjectUpdateInfo):
        cfg = update_info.handler_configs.get("fusesoc", {}) or {}
        project_dir = update_info.project_dir or os.path.dirname(update_info.deps_dir)

        # Filter by import list
        import_filter = cfg.get("import", "all")
        if import_filter == "all":
            dirs = [d for dirs in self._core_dirs.values() for d in dirs]
        else:
            dirs = [d for name in import_filter
                      for d in self._core_dirs.get(name, [])]

        # Include project's own .core files
        if _has_cores_nonrecursive(project_dir):
            dirs = [project_dir] + dirs

        if not dirs:
            return

        deps_dir = update_info.deps_dir
        _write_cores_txt(os.path.join(deps_dir, "fusesoc-cores.txt"), dirs, project_dir)
        _write_cores_envrc(os.path.join(deps_dir, "fusesoc-cores.envrc"), dirs)
        _append_to_packages_envrc(deps_dir)

        if cfg.get("update-conf", False):
            _patch_fusesoc_conf(os.path.join(project_dir, "fusesoc.conf"), dirs)

    def _get_core_dirs(self, pkg) -> List[str]:
        """Return core dirs from explicit declaration or auto-detection."""
        patterns = self._get_declared_patterns(pkg)
        if patterns is not None:
            return self._resolve_patterns(pkg.name, pkg.path, patterns)
        # Auto-detect: non-recursive scan of package root only
        if _has_cores_nonrecursive(pkg.path):
            return [pkg.path]
        return []

    def _get_declared_patterns(self, pkg) -> Optional[List[str]]:
        """Return explicit cores patterns from dep's own ivpm.yaml, or None."""
        if pkg.proj_info is None:
            return None
        cfg = pkg.proj_info.handler_configs.get("fusesoc", {}) or {}
        patterns = cfg.get("cores", None)
        return [str(p) for p in patterns] if patterns is not None else None

    def get_state_entries(self) -> dict:
        return {"core_dirs": {k: v for k, v in self._core_dirs.items()}}
```

`_has_cores_nonrecursive(path)` checks for `*.core` files in the immediate
directory only (does not recurse), consistent with the intent that package
authors explicitly declare nested core directories.

## Part 2 — `fusesoc` Source Type (New Package Type)

### What it does

Allows users to declare FuseSoC core dependencies by VLNV rather than by
raw git URL:

```yaml
package:
  name: my_soc
  with:
    fusesoc:

  dep-sets:
  - name: default
    deps:
    - name: wb_intercon
      src: fusesoc
      vlnv: "::wb_intercon:1.0"

    - name: uart16550
      src: fusesoc
      vlnv: "::uart16550:1.5"
```

IVPM resolves the VLNV by:

1. Locating the `fusesoc-cores` index (a git repo, declared as an IVPM dep
   with `cache: true`, or auto-fetched to `~/.cache/ivpm/fusesoc-cores/`).
2. Scanning index `.core` files to find one whose `name:` field matches.
3. Extracting the `provider:` block (git URL + version/tag).
4. Fetching the repository exactly as `PackageGit` would.

### VLNV Resolution Algorithm

```
1. Parse the requested VLNV (vendor:library:name:version).
2. Walk the index directory for *.core files.
3. Parse each; compare its CAPI=2 "name:" field to the VLNV.
4. On match, read provider block:
     github provider: https://github.com/{user}/{repo}.git  @  {version}
     git provider:    provider.repo                          @  provider.version
5. Return a PackageGit with that URL and version.
```

### Implementation Sketch

```python
@dc.dataclass
class PackageFuseSoC(Package):
    vlnv: str = None

    def update(self, update_info: ProjectUpdateInfo) -> ProjInfo:
        git_pkg = self._resolve_to_git(update_info)
        return git_pkg.update(update_info)

    def _resolve_to_git(self, update_info) -> PackageGit:
        index_dir = self._find_index(update_info)
        url, version = _vlnv_lookup(self.vlnv, index_dir)
        return PackageGit(name=self.name, url=url, tag=version)
```

## Integration Summary

```
ivpm.yaml
─────────
package:
  name: my_soc
  with:
    fusesoc:                    ← activates PackageHandlerFuseSoC
      import: all               ← (default)
      update-conf: false        ← (default; opt-in to touch fusesoc.conf)

  dep-sets:
  - name: default
    deps:
    - name: wb_intercon
      src: fusesoc              ← PackageFuseSoC: resolved by VLNV (optional)
      vlnv: "::wb_intercon:1.0"

    - name: my_other_lib
      url: https://github.com/my-org/my-lib.git   ← plain git also works
```

```
After `ivpm update`:
─────────────────────
packages/
  wb_intercon/          ← fetched (via VLNV → git clone)
    wb_intercon-1.0.core
  my_other_lib/         ← fetched (plain git clone)
    my_other_lib.core

  fusesoc-cores.txt     ← one core dir per line (for scripting)
  fusesoc-cores.envrc   ← export FUSESOC_CORES=...
  packages.envrc        ← (from direnv handler) + source_env ./fusesoc-cores.envrc

project .envrc          ← user-maintained, typically just:
  source_env packages/packages.envrc
```

Developers run `fusesoc run --target sim ::my_design:0.1` and all cores are
immediately available — `FUSESOC_CORES` is set by `.envrc` with no manual
configuration.

## Implementation Phasing

### Phase 1 — `PackageHandlerFuseSoC` (high value, self-contained)

- New file: `src/ivpm/handlers/package_handler_fusesoc.py`
- Register entry point in `pyproject.toml`:
  ```toml
  fusesoc = "ivpm.handlers.package_handler_fusesoc:PackageHandlerFuseSoC"
  ```
- Add tests: `test/unit/test_fusesoc_handler.py`

Deliverable: projects that already fetch HDL git deps via IVPM get auto-generated
`fusesoc-cores.envrc` and `fusesoc-cores.txt` in `packages/`, with automatic
contribution to `packages.envrc` when the direnv handler is also active.

### Phase 2 — `PackageFuseSoC` source type (nice to have)

- New file: `src/ivpm/pkg_types/package_fusesoc.py`
- Register in `PkgTypeRgy` under the name `"fusesoc"`.
- Add the fusesoc-cores index auto-fetch / cache logic.
- Add tests: `test/unit/test_pkg_fusesoc.py`

Deliverable: users can declare `src: fusesoc / vlnv:` and skip manual URL lookup.

## Design Decisions and Rationale

| Decision | Rationale |
|---|---|
| `fusesoc-cores.envrc` in `packages/` rather than the project root | Lives next to other IVPM outputs; gitignored naturally; clearly machine-generated |
| Non-recursive auto-detection of `.core` files | Avoids scanning large repos; package authors explicitly declare nested core dirs |
| Three output artifacts (`.txt`, `.envrc`, `packages.envrc` contribution) | Covers all use-cases: scripting, direnv, manual shell setup |
| `update-conf: false` default | `fusesoc.conf` is user-maintained; `configparser` has no include support so any write risks surprise; `.envrc` path is recommended |
| Phase 10 for FuseSoC handler (direnv is phase 0) | Ensures `packages/packages.envrc` exists before FuseSoC appends to it |
| Publish/import model mirrors agents handler | Consistent IVPM idiom; package authors control what they expose; consumers control what they import |
| VLNV source type is optional | Plain git + auto-detect covers most cases; VLNV lookup adds convenience, not necessity |
| Index as a normal IVPM `cache: true` dep | Leverages existing caching; no new cache layer needed |

## Open Questions

1. **VLNV version matching**: When multiple `.core` files exist for the same
   name at different versions, should IVPM pick the latest, closest match, or
   error? Recommendation: strict match, error if not found, with a
   `version: latest` shorthand.

2. **Transitive VLNV deps**: FuseSoC resolves inter-core dependencies at build
   time. Should IVPM recursively fetch VLNV deps at `ivpm update` time?
   Recommendation: no — let FuseSoC handle core-to-core dep resolution; IVPM
   only fetches what the user explicitly lists.

3. **Index freshness**: How often is the auto-fetched index updated?
   Recommendation: honour `cache: true` semantics (stale after a configurable
   TTL, refreshed on `ivpm update --refresh`).

4. **`packages.envrc` append safety**: If the direnv handler regenerates
   `packages.envrc` after the FuseSoC handler has appended to it, the
   `source_env ./fusesoc-cores.envrc` line would be lost. Resolution: the
   direnv handler should be extended with a `get_envrc_lines() -> List[str]`
   hook on `PackageHandler` so contributions are collected during direnv's own
   `on_root_post_load` rather than appended after the fact.
