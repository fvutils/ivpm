# FuseSoC Examples for IVPM

## Overview

This document describes two complementary examples that showcase IVPM's FuseSoC integration:

1. **Registry Example** — Fetches a FuseSoC core from the public registry using VLNV addressing
2. **Custom Example** — Clones a custom Git repository containing a FuseSoC-enabled IP

Both examples demonstrate the complete workflow: `ivpm update && fusesoc <action>`.

- **Simplicity**: Keep examples minimal so the core concepts are clear

- **Educational**: Clearly explain what IVPM is doing at each step

## Background: FuseSoC Concepts

### VLNV Addressing
FuseSoC identifies cores using **VLNV** (Vendor:Library:Name:Version) strings:
```
::wb_intercon:1.0
```
The registry at [fusesoc/fusesoc-cores](https://github.com/fusesoc/fusesoc-cores) maps VLNVs to git repos and versions.

### Core Files
HDL designs are described in `.core` files (CAPI=2 YAML format):
```yaml
CAPI=2:
name: "vendor:lib:corename:version"
filesets:
  rtl:
    files:
      - rtl/module.v
targets:
  default:
    filesets: [rtl]
```

### FuseSoC as a Tool
FuseSoC is a Python package that:
- Discovers `.core` files via `FUSESOC_CORES` environment variable
- Resolves dependencies and manages tool configurations
- Runs simulations, builds, or generates project files

## Example 1: Registry-Based FuseSoC Core

### Purpose
Demonstrate resolving a FuseSoC core from the public registry using VLNV, installing FuseSoC tools, and running a basic operation.

### Directory Structure
```
ex1-registry/
├── README.md              # Explanation and usage
├── .envrc                 # direnv file (sources packages/packages.envrc)
├── ivpm.yaml              # IVPM configuration
└── packages/              # Created by ivpm update
    ├── python/            # Python venv with FuseSoC
    ├── i2c/               # Fetched i2c core (from VLNV lookup)
    ├── verilator/         # Verilator simulator binary
    ├── packages.envrc     # Generated by direnv handler
    └── fusesoc-cores.envrc  # Generated by fusesoc handler
```

### ivpm.yaml Structure
```yaml
package:
  name: ivpm-fusesoc-example-registry
  
  dep-sets:
  - name: default
    deps:
    # FuseSoC tool (Python package from PyPI)
    - name: fusesoc
      src: pypi
    
    # Verilator simulator (optional, for running simulations)
    - name: verilator
      src: gh-rls
      url: https://github.com/edapack/verilator-bin
    
    # i2c core from the FuseSoC registry
    - name: i2c
      src: fusesoc
      vlnv: "::i2c:1.0"

  with:
    # Set up Python venv with FuseSoC
    python: {}
    
    # Generate FUSESOC_CORES and fusesoc.conf
    fusesoc:
      import: [i2c]               # Only import HDL cores, skip tool packages
```

### .envrc File
```bash
# Load IVPM-managed environment (Python venv + FuseSoC setup)
source_env packages/packages.envrc
```

### Key IVPM Features Showcased
1. **Python Handler**: Sets up a managed Python venv in `packages/python/`
2. **FuseSoC Source Type**: Resolves `vlnv: "::i2c:1.0"` to a git URL via the registry
3. **FuseSoC Handler**: Discovers `.core` files and generates `FUSESOC_CORES` environment

### Usage Flow
```bash
cd ex1-registry
ivpm update          # Fetch FuseSoC, i2c core, and registry index
. packages/packages.envrc  # Source Python venv and FuseSoC setup
fusesoc core list    # Show available cores
fusesoc lint i2c     # Validate the i2c core
```

### Documentation Sections
1. **What IVPM Did**
   - Created Python venv with FuseSoC installed
   - Resolved `::i2c:1.0` VLNV via fusesoc-cores registry
   - Cloned the i2c repository into packages/i2c/
   - Generated `FUSESOC_CORES` environment pointing to the i2c directory

2. **What You Can Do Next**
   - `fusesoc core list` — See all available cores
   - `fusesoc core show ::i2c:1.0` — Inspect the i2c core description
   - `fusesoc lint ::i2c:1.0` — Validate the core file
   - (Advanced) Check `packages/fusesoc-cores.txt` to see core discovery

3. **Key Takeaway**
   - IVPM + FuseSoC VLNV = declarative IP management
   - No manual `fusesoc.conf` editing needed

## Example 2: Custom Git Repository with FuseSoC IP

### Purpose
Demonstrate a custom Git repository containing multiple `.core` files, showing how IVPM auto-discovers cores and enables FuseSoC workflows.

### Directory Structure
```
ex2-custom/
├── README.md              # Explanation and usage
├── .envrc                 # direnv file (sources packages/packages.envrc)
├── ivpm.yaml              # IVPM configuration
└── packages/              # Created by ivpm update
    ├── python/            # Python venv with FuseSoC
    ├── uart16550/         # Example: UART core (or custom project)
    ├── verilator/         # Verilator simulator binary
    ├── packages.envrc     # Generated by direnv handler
    └── fusesoc-cores.envrc  # Generated by fusesoc handler
```

### ivpm.yaml Structure
```yaml
package:
  name: ivpm-fusesoc-example-custom
  
  dep-sets:
  - name: default
    deps:
    # FuseSoC tool
    - name: fusesoc
      src: pypi
    
    # Verilator simulator (optional, for running simulations)
    - name: verilator
      src: gh-rls
      url: https://github.com/edapack/verilator-bin
    
    # Example: Use an existing FuseSoC-enabled project
    # (Replace with a real project URL or use a local path for testing)
    - name: uart16550
      src: fusesoc
      vlnv: "::uart16550:1.0"

  with:
    # Set up Python venv with FuseSoC
    python: {}
    
    # Auto-discover and register .core files
    fusesoc:
      import: all  # Include all packages' cores
```

### .envrc File
```bash
# Load IVPM-managed environment (Python venv + FuseSoC setup)
source_env packages/packages.envrc
```

**Note on Custom Projects:**
This example uses `uart16550` from the FuseSoC registry as a substitute. In production, you would replace this with your own custom project URL (Git repo or local path) that contains `.core` files. The IVPM FuseSoC handler will auto-discover cores in any package, regardless of source.

### Key IVPM Features Showcased
1. **Auto-Discovery**: FuseSoC handler scans packages for `.core` files (no annotation needed)
2. **Environment Setup**: Generates `FUSESOC_CORES` pointing to the custom repo
3. **State Persistence**: Stores discovered cores in `ivpm.json` for incremental updates

### Usage Flow
```bash
cd ex2-custom
ivpm update                 # Fetch FuseSoC and custom repo
. packages/packages.envrc   # Activate Python venv + FuseSoC setup
fusesoc core list           # Show available cores (from my-hdl-lib)
fusesoc core show ::my_hdl_lib::module_a  # Inspect module_a
# (If sim.core has simulation targets)
fusesoc run ::my_hdl_lib::module_a:1.0 --target=sim
```

### Documentation Sections
1. **What IVPM Did**
   - Created Python venv with FuseSoC installed
   - Cloned my-hdl-lib into packages/my-hdl-lib/
   - Scanned my-hdl-lib/ for `.core` files (found rtl.core and sim.core)
   - Generated `FUSESOC_CORES` pointing to packages/my-hdl-lib/
   - Wrote `fusesoc.conf` with the custom library registered

2. **What You Can Do Next**
   - `fusesoc core list` — See all available cores in the custom repo
   - `fusesoc lint ::my_hdl_lib::*` — Validate all cores
   - Modify IP in packages/my-hdl-lib/ and re-run `fusesoc` commands

3. **Key Takeaway**
   - IVPM + FuseSoC auto-discovery = declarative HDL library management
   - No per-package annotations needed (default behavior)
   - fusesoc.conf is auto-maintained by IVPM

## Shared Features Across Both Examples

### Environment Setup (packages/packages.envrc)
After `ivpm update`, both examples source `packages/packages.envrc`, which:
```bash
# Python venv activation (from python handler)
source $(pwd)/packages/python/venv/bin/activate

# FuseSoC core directories (from fusesoc handler)
export FUSESOC_CORES="/abs/path/packages/i2c:/abs/path/packages/my-hdl-lib"
```

### IVPM State Files
Both examples generate:
- `packages/fusesoc-cores.txt` — Machine-readable core directory listing
- `packages/fusesoc-cores.envrc` — Shell-sourceable FUSESOC_CORES export
- `ivpm.json` — State snapshot (includes discovered cores, handler state)

### Directory Independence
Each example is **self-contained**:
- No shared state between ex1 and ex2
- Each can be run independently
- Both follow the same `ivpm update && fusesoc <action>` pattern

### Custom Repository Option
Example 2 uses `uart16550` (an existing FuseSoC-enabled project) instead of requiring a custom repository to be created. This keeps the examples self-contained and easy to set up.

- A URL to your custom Git repository containing `.core` files, or
- A local path to an existing HDL library

The IVPM FuseSoC handler automatically discovers and registers all `.core` files in any fetched package.
